# Tension 張力：同一個系統，不同角度的 Value

---

「統一資料格式」和「忠實保留來源」都是對的。「回應今天的需求」和「保留未來的可能」都是對的。兩個都對的東西互相拉扯，這叫**張力 (tension)**。

這是概念基礎系列的第三篇。[第一篇](./emergence-data-compute.md)講時間軸（什麼留下來），[第二篇](./compute-state-context.md)講當下（state 如何遇見 compute），這一篇講**視角**：同一個系統，不同角度看見不同的 value——而本文想說服你，張力不是待解決的問題，是系統的**結構**。

---

## 1. 張力不是 Bug

先分清楚兩個常被混用的詞：

- **矛盾**：有一方是錯的。找出來，修掉。
- **張力**：**兩方都對**，而且互相拉扯。

矛盾該被消除，張力不該——吊橋靠張力站立，把纜索的張力「解決」掉，橋不是變好了，是塌了。

遇到張力時最常見的錯誤反應，是急著讓它消失：急著選邊、急著折衷、急著宣布「我們已經對齊了」。三種做法都一樣——用消滅一邊的 value 來換取表面的安靜。

---

## 2. 事實單一，價值多元

這是全篇的錨點。

我們的 Vault 叫 **Single Source of Truth**——事實應該收斂成一個。但 **truth 單一，不代表 value 單一**：

> 事實回答「是什麼」；價值回答「對誰重要」。
> 前者可以收斂成一個，後者永遠不該收斂成一個。

同一筆產品資料：定價分析看到價格帶、成分研究看到配方、品牌策略看到鋪貨廣度。三種 value，零矛盾——它們只是站在不同的角度。

**13 個維度就是 13 個角度。** 維度不是資料的屬性，是「看資料的方式」。這也是 MDFO 的設計哲學：Measure × Dimension 的自由組合，本質上是**讓提問者自己選角度**——系統不預設哪個角度有價值。

---

## 3. 系統如何讓多視角活著

這不是空談，架構裡已經這樣做了：

- **MDFO 是一台多視角機器**：同一個資料庫，換一組 Measure 和 Dimension，就是另一個世界的問題。
- **TheDistiller 的兩手**：Identity Resolution 把多來源統一成 canonical 身分，但**原始來源紀錄都留著**——「統一」疊在「忠實」之上，不覆蓋它。統一 vs 忠實這條張力，答案不是二選一，是讓兩邊都活著。
- **Factum 的分級路線**：品質好的圖走便宜的快路線，品質差的走昂貴的深度路線——「完整」與「及時」的張力用**分級**共存，而不是全面妥協。

三個案例的共同點：**張力被保留在結構裡，而不是被一個「正確答案」抹掉。**

---

## 4. 下棋：看得見張力的人，控制遊戲

棋手有一個術語叫「保持張力 (maintaining the tension)」。

兩隻兵互相咬住、都能吃掉對方，但誰都還沒吃——這就是棋盤上的張力。新手最常犯的錯是**急著兌掉**：吃掉它，局面清爽了，心裡舒服了。但兌掉的那一刻，你也放棄了所有選項。

高手反著做。他**維持張力**，因為：

1. **張力保留選項**——只要不兌，「吃、不吃、什麼時候吃」都還是你的牌。
2. **張力消耗對手**——對手每一步都得重新計算那個懸而未決的交換，你把計算負擔丟給了他。
3. **由你決定何時釋放**——最後兌掉的時機、方式、代價，由看得清張力的那一方決定。

**看得見真實張力的人，控制遊戲。** 看不見的人只感覺到「不舒服」，然後急著讓不舒服消失——那一刻他就被操控了。

這正是我們的系統要給使用者的東西：市場本身充滿張力（成分在流行但價格帶在崩、劑型在轉移但產能沒跟上）。**把市場的張力攤開給看得懂的人，他就拿到了決定「何時釋放」的權力。** 我們賣的不是資料，是棋感。

我們自己的棋盤上也有幾組長期張力：

| 張力 | 一邊 | 另一邊 | 我們的走法 |
|---|---|---|---|
| 今天 vs 未來 | 回應眼前需求 | 保留湧現空間 | 蒐集邊界從寬、raw 永遠留（[第一篇](./emergence-data-compute.md)） |
| 統一 vs 忠實 | canonical 資料 | 來源原貌 | 統一層疊在原始層之上，不覆蓋 |
| 完整 vs 及時 | 爬好爬滿、解析到底 | 先交付 | 品質分級路由：好走快路、差走深路 |
| 簡單 vs 精確 | 業務語言提問 | 完整表達力 | MDFO 用業務語言，複雜度由引擎吸收 |

注意每一行的「走法」都不是選邊——都是**讓兩邊繼續存在的結構**。

---

## 5. 人的張力：B 端不是一種人

前面講的是資料與設計的張力。還有一層更重要的：**人的張力**——這正是我們的系統存在的理由。

我們的系統要讓 **B 端看到 C 端**。但「B 端」不是一種人，同一個 C 端市場，每種角色看到的是完全不同的東西：

| 角色 | 看同一個 C 端市場時，看到的 value |
|---|---|
| **原料供應商** | 我的成分出現在多少產品裡？劑量帶在哪？這個成分在成長還是衰退？ |
| **品牌商** | 競品的定位與價格帶、消費者的劑型偏好、市場的缺口在哪？ |
| **OEM / ODM** | 市場在往什麼配方技術、什麼劑型走？我該準備什麼產線？ |
| **學術端** | 市場實態作為研究資料：成分與健康宣稱之間的實證縫隙在哪？ |
| **監管端** | 標示合規嗎？宣稱有沒有越界？市場上實際流通的是什麼？ |

同一份事實，至少五種 value。**系統不能偏袒任何一個角度**——為品牌商優化的視角會讓原料商看不到它要的東西；被消除的視角不會抗議，它只是安靜地流失成別人的客戶。

而且這些角色**彼此之間**也在張力中：原料商希望成分透明，品牌商需要配方保密；監管端要宣稱保守，品牌端要行銷張力；學術端要長期資料，商業端要即時快照。回到下棋——**能同時看見整個棋盤上所有張力的，只有平台。** 這就是我們的位置，也是我們的護城河：每個角色只看得到自己那一角的張力，我們看得到全局。

---

## 6. 守則

1. **遇到張力，先讓兩邊的 value 都被看見，再做決定。** 消除視角是最容易的事，也最不可逆——被消除的視角不會抗議，它只是安靜地消失。
2. **做了取捨，留下紀錄。** 取捨本身是 state（[第二篇](./compute-state-context.md)的鐵律：藏在腦子裡的 state 會隨人消失）。寫下當時看見的兩邊 value 與選擇的理由，未來的人才能重新評估。
3. **不要急著釋放張力。** 「懸而未決」的不舒服是正常的；急著讓它消失的那一方，通常就是被操控的那一方。
4. **警惕只剩一種聲音的舒適感。** 一個沒有張力的會議、一份沒有反方的分析、一個所有人都同意的設計——多半不是對齊了，是有視角已經死了。

---

## 7. 對位

有一種寫音樂的方法：讓好幾個聲部同時進行，各自成立、互相牽制，誰也不服從誰。聽的人若只追任何單一聲部，都會覺得它好像在跟別人打架——但退一步聽整體，那些互相拉扯的線條合起來，是單一旋律永遠到不了的地方。

好聽的不是任何一個聲部。**好聽的是張力本身。**

三百年前就有人把這件事做到了極致。
